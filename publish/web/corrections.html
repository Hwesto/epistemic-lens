how we fix what we got wrong

If you found an error

Three classes of error get treated differently:

1. Hallucinated quote (a quote that doesn't appear in the article)

This is the most serious class. The validator (analytical.validate_analysis) checks every quote in every analysis against the source corpus before publication, but validation has gaps. If you find one we missed:

  • Open an issue at GitHub → Issues → Hallucinated quote with the date, story_key, the quote text, and the URL of the source article.
  • We retract the quote within 24 hours by editing the canonical analyses/<DATE>_<story>.json, re-render the markdown, and publish a correction note in corrections.json (linked below).

2. Number reconciliation error

An analysis cites a count or share that doesn't match the metrics.json for that story. Same channel, same SLA. Number errors are rarer than quote errors because the validator's number-reconciliation pass catches most of them.

3. Framing call you disagree with

This is not a correction request — it's editorial criticism, which we welcome but handle differently. The framing analyses are produced by Claude under the prompt at .claude/prompts/daily_analysis.md, constrained to the closed 15-frame Boydstun/Card codebook (frames_codebook.json). If you think a frame call is wrong, please use the methodology-challenge intake rather than the corrections channel.

Published corrections log

Each retraction is recorded in corrections.json with the date, the original claim, the corrected claim, and the source evidence. The log is append-only.

What does NOT get corrected

We do not retract published analyses because:

  • The framing changed in our subjective view. Analyses are stamped to meta_version; readers can read them as "what the system said on day X" not "what's true today."
  • A source you don't like was quoted. The source-attribution layer is descriptive — who appeared in the corpus — not prescriptive.
  • A bucket was missing or uneven. The bucket-quality tier system in bucket_quality.json handles this transparently; EXCLUDE_QUANT buckets are quarantined from quantitative metrics automatically.
