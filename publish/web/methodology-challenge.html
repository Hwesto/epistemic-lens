METHODOLOGY CHALLENGE

how to argue with the framing analyses

Framing analyses are subjective by construction. The closed codebook smooths real per-event variation, the 1500-char cap on signal text affects what gets analysed, the bucket grouping decides what counts as "the same story." If you disagree with a framing call, the project treats that as a methodology challenge, not a correction request.

The methodology you're challenging

  • Codebook: frames_codebook.json (15 frames, closed; descended from Boydstun/Card Media Frames Corpus).
  • Prompt: .claude/prompts/daily_analysis.md (the instruction Claude follows; pinned by meta_version).
  • Pipeline: docs/METHODOLOGY.md.
  • Bucket grouping: feeds.json (235 feeds across 54 buckets).

Where to file a challenge

Open an issue at GitHub → Issues → Methodology challenge. The template asks for:

  1. The specific claim you're challenging (date + story_key + frame_id).
  2. The methodology element you think is at fault (codebook, prompt, bucket grouping, signal-text cap, etc.).
  3. What the alternative call would have been, and why.
  4. Whether you think this is one-off (this story, this day) or systemic (the codebook genuinely doesn't accommodate your reading).

How challenges are resolved

  • One-off. Logged in methodology_challenges.json; no edit to the published analysis. Future analyses unchanged.
  • Systemic. The challenge becomes a candidate for the next major meta_version bump. Codebook revisions and prompt rewrites are major-bump territory because they invalidate longitudinal comparison.
  • Validation-set evidence. Phase 4 of the project will hand-code 100 articles for ground truth; if a challenge is backed by hand-code disagreement with the analyzer's call, it carries more weight in the next codebook revision.

Distinct from a correction: a correction targets a factual error (hallucinated quote, wrong number); a challenge targets a methodological choice. We treat them with different SLAs because they need different remedies.
